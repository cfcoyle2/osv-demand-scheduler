# Library Naming Conventions

Use these conventions to keep folders and files consistent across the Library.

## Folder Names

- Use `Title_Case_With_Underscores` for project folders.
- Keep names short and specific.
- Use category-first folders only at the top level of the Library.

Examples:

- `Sparta_EV_Runs`
- `Logistics_Lookahead`
- `Kabal`
- `Project_Work_Package_Template`

## File Names

- Use clear descriptive names.
- For working documents, prefer `Title_Case_With_Underscores`.
- Keep the final file extension standard and visible.
- Avoid spaces for new working files when possible.

Examples:

- `Manager_Brief.md`
- `Pros_Cons_Table.md`
- `Run_Schedule_Analysis.xlsx`
- `Weekly_Lookahead_Draft.pptx`

## Website Files

- Use `index.html` as the default launch page inside a website folder.
- Keep site support files in `assets/`, `docs/`, and `data/`.
- Use lowercase names for web assets when practical.

Examples:

- `assets/css/styles.css`
- `assets/js/app.js`
- `data/lookahead.json`

## Scripts

- Use verb-first names for scripts.
- Keep PowerShell and Python scripts in their language folders under `05_Scripts_and_Automation` unless they are project-specific.

Examples:

- `build_sparta_ev_deck.ps1`
- `extract_pdfs.py`
- `generate_weekly_summary.ps1`

## Versioning

- Avoid adding version numbers to active working files unless required.
- For milestones or archived versions, append a clear suffix.

Examples:

- `Sparta_EV_Run_Strategy_Draft.pptx`
- `Sparta_EV_Run_Strategy_Final.pptx`
- `Weekly_Lookahead_Archive_2026_05.pptx`

## Recommended Rule of Thumb

If a file will be reused, shared, or revisited later, name it so someone can understand its purpose without opening it.
